1 	Introduction
Filien est un programme batch de création automatique de prévisions, de pré-engagements,
d'engagements, de pré-mandats ou de pré-recettes dans l'application GF SEDIT Marianne à partir
du contenu d'un fichier Ascii respectant un format prédéfini.
Il permet également de faire passer à une phase supérieure un mouvement déjà existant (exemple :
mandater un mouvement engagé précédemment) à condition de positionner l'information /10/ à "O".
Ce programme autorise la création de réductions et d'annulations, mais pas celle de rejets de
mandats et de titres.
Il permet la saisie de P503, de débits d'office, de mandats collectifs... à condition de positionner
correctement le code nature Indigo (/19/).
Ce programme ne permet pas la création de pré-bordereau de reprise (type de pré-bordereau = "R")
ni la saisie d'avoirs (mouvements de type "A").

1.1 	Présentation du fichier Ascii
 Séparateur entre 2 mouvements : /##/. Il doit être placé à la suite de l'en-tête et du dernier
enregistrement du fichier.
 Séparateur entre 2 lignes d'un même mouvement : /--/. Il est à insérer également devant la 1ère
ligne de mouvement, mais pas après la dernière).
 Séparateur entre 2 détails de prestations pour les ASAP Dématérialisés : /**/.
 Les informations /500/ à /509/ sont réservées aux recettes. Elles peuvent apparaitre plusieurs
fois pour un même titre, à condition d'être séparées par /**/.
 Séparateur entre 2 tiers solidaires sur un même titre : /$$/.
 Les informations /400/, /401/ et /402/ sont réservées aux recettes qui ont des tiers solidaires
(balise /183/O). Elles peuvent apparaître plusieurs fois dans un titre et sont séparées par /$$/.
 Séparateurs indiquant la gestion d'un lot (ensemble de mouvements à traiter dans une seule
transaction) : {DEBUT_LOT} et {FIN_LOT}. Ils sont à insérer avant le séparateur /##/.
 Les informations /11/ à /49/ et /71/ à /99/ sauf les informations /39/ et /391/ sont réservées aux
mandats et recettes. Les engagements ne doivent être décrits qu'à partir d'informations /01/ à
/10/, /39/ et /391/ et /51/ à /69/.
 Les informations /51/ à /99/ peuvent apparaître plusieurs fois pour un même mouvement à
condition d'être séparées par /--/.
 Les informations /71/ à /99/ sont plus particulièrement destinées à la prise en compte des
mandats et titres de recette collectifs.
 Il n'est pas obligatoire de saisir les données dans l'ordre numérique croissant.
 Si une même zone comporte plusieurs informations (exemples : domiciliation /39/, code
e-Marché /52/, imputation /54/, /541/, /542/...) il est impératif de respecter l'emplacement de
chaque information en complétant éventuellement la chaîne de caractères avec des espaces.
Exemple : Soit le service gestionnaire RH et destinataire FINANCE sur la segmentation
sectorielle codée STRUCT_1. Cette information doit être transmise de la façon suivante dans le
fichier texte : /542/STRUCT_1..RH........FINANCE (les points symbolisent les espaces laissés entre
les 2 informations).
 Il n'est pas nécessaire de compléter avec des zéros à gauche les zones Montant. La virgule (,) est
le seul séparateur accepté dans une zone Montant.
1.2 	Évolutions 2022.1
Le FILIEN permet de prendre en compte la proratisation de la TVA.
Aucune nouvelle balise n'a été ajoutée.
1.3 	Évolutions 2021.1
La version 2021.1 permet de référencer une opération sur un engagement (mouvement) ou un mandat
(balise /544/). Elle accepte la référence à un RIB/IBAN sur la création d’un engagement (mêmes balises
que pour les pré-mandats). La balise (/505/) quantité pour les prestations sur les ASAP dématérialisés
est revue au niveau de la longueur et du format.

1.4 	Évolutions 2019.1
La version 2019.1 permet de définir un titre avec des tiers solidaires et de donner une liste de tiers
solidaires. La balise /183/ permet de préciser que le titre comporte des tiers solidaires en précisant la
valeur O pour Oui ou N par Non. Si la balise n’existe pas, cela génère un titre sans tiers solidaires.
Pour donner les informations nécessaires sur le(s) tiers solidaire(s), les balises /400/, /401/, /402/
permettent de préciser la liste de ces tiers solidaires.
1.5 	Évolutions 2018.1.04
Au lancement de l'interface FILIEN, un nouveau paramètre Génération d'ASAP dématérialisés
(prestation globale) permet de créer les détails des prestations d'un ASAP dématérialisé à partir des
lignes de titres, comme en saisie des liquidations. Il n'est pas nécessaire de cocher cette case lorsque le
filien contient les balises nécessaires aux ASAP (balises /500/ ...). Cette case est utile lorsque le filien
génère des titres qui doivent être des ASAP dématérialisés et que l'utilitaire de génération du FILIEN ne
crée pas les balises correspondantes.
1.6 	Évolutions 2018.1
La version 2018.1 permet d'alimenter le type de document de chaque pièce jointe liée aux écritures
injectées. Les balises /266/, /276/, /286/, /296/, /306/ permettent de donner ces informations pour les PJ
de la n°1 à la n°5.
Dans l'interface FILIEN exécuté depuis le Web, il est possible d'indiquer un type de document qui est
appliqué par défaut à toutes les PJ qui n'ont pas une des balises ci-dessus.
Les balises /262/, /272/, /282/, /292/, /302/ permettent de spécifier une ou des pièces en GED,
conjointement avec les balises /263/, /273/, /283/, /293/, /303/ qui permettent dans ce cas de spécifier
l'adresse (url) du document GED correspondant.
La version 2018.1 propose une nouvelle balise /182/ qui permet d’indiquer une date de demande de
paiement. Cette date sera ensuite transmise dans le flux PES au niveau de la balise DteEch, elle est
obligatoire pour les mandats de prélèvement à la source (PAS).
1.7 	Évolutions 2017.2
La version 2017.2 permet d'alimenter la balise 585 avec le numéro de rôle pour le titre récapitulatif.
1.8 	Évolutions 2017.1
La version 2017.1 permet d'alimenter le détail des prestations prévues dans la saisie des recettes afin
de dématérialiser les avis des sommes à payer (ASAP) avec 10 nouvelles zones (de /500/ à /509/). Les
informations Chorus Pro nécessaires pour les ASAP des tiers de la sphère publique peuvent aussi être
alimentées afin d'être transmises par la suite sur le portail Chorus Pro (de /441/ à /445/).

1.11 	Maintenance 2014.1.01 / 2013.1.09
Les versions 2014.1.01 et 2013.1.09 n'introduisent pas de nouvelles balises mais modifient les règles de

(IBAN) afin de les conformer aux règles d'Hélios en matière de modes de règlement.

1.12 	Évolutions 2013.1.05
La version 2013.1.05 apporte une série d'évolutions au protocole filien visant l'atteinte de trois objectifs
:
 La comptabilité avec le PES : de nouvelles balises permettent de référencer des pièces justificatives
et de désigner les informations auxiliaires de la ligne (pièce de rattachement, numéro d'emprunt...).
Par ailleurs, de nouvelles règles de gestion s'appliquent aux civilités et prénoms afin d'assurer la
cohérence du tiers particulier.
 L'intégration de titres individuels pour prélèvement automatique SEPA. Cette nouvelle forme de titre
de recette implique que la collectivité exploite le PES. Elle nécessite notamment de valoriser les
balises dédiées au RUM et à l'ICS.
 La comptabilité avec le module eCadre budgétaire (dès la version 2013.1.04).
Pièces justificatives
Pour les collectivités exploitant le PES, les balises /261/ et suivantes sont dédiées à l'import des pièces
justificatives. Il peut y avoir jusqu'à 5 PJ par mandat ou titre.
Trois balises permettent d'identifier chaque PJ :
 un libellé obligatoire,
 un type de support obligatoire : il peut prendre les valeurs 01 électronique, 02 papier ou 03
DVDrom,
 un raccourci : il est obligatoire si le type de support est 01 et interdit sinon.
Dans le cas général, avec l'intégration du fichier Filien, l'application créera les pièces justificatives en
leur attribuant le numéro d'identifiant unique par incrémentation de leur compteur.
Dans le cas où il existe déjà une PJ avec les mêmes caractéristiques (c'est-à-dire même budget, même
exercice, même libellé, même type de support et même chemin_fichier ou absence de chemin_fichier)
alors la PJ n'est pas créée et le mandat reçoit la référence à celle déjà existante.
La version 2015.3 introduit deux nouvelles balises pour chacune de PJ. Elles permettent de
désigner le type de pièce (264) et le format (265).
eCadre budgétaire
Avec le module eCadre budgétaire, les balises /541/ à /545/ succèdent à désignation de l'imputation
étendue par l'information /54/. Ces balises /541/ à /545/ doivent être renseignées dès lors qu'il existe
une "modalité de saisie - Exécutions" pour le budget et l'exercice identifiés par le filien. Les zones
obligatoires définies sur la modalité de saisie doivent être respectées.

1.13 	Mandats et titres de recette collectifs
L'enregistrement des mandats et titres collectifs doit suivre la procédure suivante :
 Créer un entête de mandat en indiquant dans l'information /31/ le nom du débiteur ou du
créditeur divers.
 Renseigner le code Titre collectif (/44/) à 'O'.
 Créer autant de lignes de mouvements que nécessaire. L'information N° de ligne de mouvement
(/51/) doit conserver la même valeur d'une ligne sur l'autre. Seul le N° dans le rôle (/71/) doit varier
ainsi que les références des tiers. De ce fait, un titre récapitulatif comportant une seule ligne sera
émis, mais le détail des tiers concernés sera transmis sur bande magnétique, grâce aux données
/71/ à /99/.

2 	Format du fichier texte Filien
2.1 	Contenu du premier enregistrement "en-tête"
/##/PARAM/a/b/c/d/e/f/g
a) Code organisme Finances (de 01 à 99 - doit correspondre à un enregistrement Orgafi)
b) Code budget Finances (sur 2 lettres - doit correspondre à un enregistrement Budget lié à
l'organisme sélectionné)
c) Exercice à traiter (sur 4 chiffres - doit correspondre à un enregistrement dans Budannu lié au
budget sélectionné)
d) Code avancement des mouvements
1 = mouvements en prévision
2 = mouvements pré-engagés
3 = mouvements engagés
4 = mouvements facturés
5 = mouvement pré-mandatés
e) Rejet si dépassement du disponible (O/N)
f) Rejet si dépassement du C.A. maxi hors marché avec un tiers (O/N)
g) Rejet si dépassement du Montant global du marché (O/N)
Exemple : /##/PARAM/01/XY/1996/5/O/N/N
L'enregistrement "en-tête" peut apparaître plusieurs fois dans le même fichier ASCII, en cas
d'engagement ou de mandatement sur plusieurs budgets.
2.2 	Mouvements déjà existants en Finances
Il est possible de mandater grâce à Filien, un mouvement préalablement engagé ou d'engager un
mouvement saisi en prévision.
Dans ce cas :
 dans l'entête du mouvement, renseignez uniquement la donnée /01/ et saisissez la lettre "O"
dans l'information /10/.
 les lignes référencées (identifiées par /51/) doivent obligatoirement exister pour le mouvement
en question. Filien ne génère pas de nouvelles lignes de mouvement pour un mouvement déjà
créé.
 les informations /52/ à /69/ sont facultatives, sauf l'information /54/ (imputation) qui est
interdite.
Le programme Filien prend par défaut les informations contenues dans le mouvement existant. Par
contre, si les informations /52/ à /69/ sont saisies, elles viendront se substituer à celles de ligne de
mouvement (avec mises à jour correspondantes en cas de changement de marché, de réservation ou
de montant).

3 	Notion de "Traitement par lot"
La notion "Traitement par lot" permet de solidariser plusieurs enregistrements afin de pouvoir les
valider ou au contraire les rejeter tous ensemble.
On peut ainsi imaginer qu'il n'y a pas d'intérêt à créer un mouvement si l'ensemble des lignes qui le
compose ne sont pas validées. En englobant dans un même lot le mouvement et toutes ses lignes, le
rejet d'une seule ligne entraînera également le rejet du mouvement complet y compris les lignes
valides.
Un lot est délimité par un ordre de début de lot : {DEBUT_LOT} et un ordre de fin de lot : {FIN_LOT}.
Cet ordre doit être seul sur une ligne du fichier Ascii. Il commence par une accolade ouvrante et se
termine par une accolade fermante. Il doit impérativement être inséré avant un séparateur de
mouvement /##/.
Tant que le programme ne rencontre pas d'ordre de début de lot, il valide ou rejette les
enregistrements traités les uns après les autres. Le rejet d'un enregistrement ne peut pas avoir d'effet
sur les enregistrements précédemment validés.
Lorsque le programme rencontre un ordre de début de lot, il ne valide les enregistrements qui suivent
que globalement, dès qu'un ordre de fin de lot est donné. Si l'un des enregistrements est rejeté, c'est
l'ensemble des enregistrements du lot qui sont rejetés, qu'ils soient situés avant ou après
l'enregistrement incriminé.
Un enregistrement de fin de lot déclenche l'opération de validation du lot (ordre commit sur la base de
données Oracle). Après cet ordre de fin de lot, si aucun ordre d'ouverture de lot n'est donné, le
programme recommence à valider ou à rejeter les lignes une à une.
Cas particuliers
 Si un ordre de début de lot a été donné mais qu'à la fin du fichier Ascii aucun ordre de fin de lot n'a
été rencontré, l'ensemble des enregistrements traités est validé (sauf rejet pour anomalie).
 Si un ordre de début de lot est donné à la suite d'un autre ordre de début de lot sans ordre de fin
de lot intermédiaire, alors le deuxième début de lot est ignoré.
 Si un ordre de fin de lot est donné alors qu'aucun ordre de début de lot n'a été préalablement
donné, alors la fin de lot est ignorée.

6 	Table de stockage des anomalies :
FILIEN_RAP
Si la table FILIEN_RAP est présente dans le compte FI de la base Oracle, le programme FILIEN y reporte
les anomalies rencontrées en cours de traitement. Chaque anomalie est identifiée par le numéro de
l'information qui a produit l'anomalie.
N° 01 	Anomalie sur le code du mouvement,
N° 03 	Anomalie sur le code Tiers,
...
Le libellé de l'anomalie, quant à lui, est le même que celui édité sur le rapport.
Le script ci-dessous correspond à la création de la table de rapport FILIEN_RAP et de l'index associé (en
précisant les tablespaces adéquats pour la table et l'index).
create table FILIEN_RAP
( id_batch varchar2(100),
section varchar2(20),
ligne number(6),
cleacces varchar2(40),
erreur number(6),
libelle varchar2(80) )
tablespace <nom tablespace des table>;
create index I0_FILIEN_RAP on FILIEN_RAP (id_batch)
tablespace <nom tablespace des index>

7 	Protocole Firetlien - Retour d'informations sur
Filien
Le programme batch Firetlien de l'application Finances c/s est accessible dans le menu Autres
travaux..., sous-menu RFL Retour Filien.
Il permet de générer un fichier texte au format décrit ci-après, contenant une liste de mouvements
générés par Filien et ayant atteint au minimum le stade Mandat validé. Les mouvements contenus
dans ce fichier texte dépendent de critères de sélection positionnés par l'utilisateur.
7.1 	Critère de sélection et de tri
Les critères de sélection des lignes de mouvement sont les suivants :
 Exercices (de ... à ...)
 Mouvements (de ... à ...)
 Nature (Dépenses et/ou Recettes)
 Avancement minimum (mandaté ou pris en compte en trésorerie)
 Dates de validation du mandat (de ... à ...)
 Dates de dernier retour Ocre (de ... à ...)
 Nom de l'utilisateur ayant créé les mouvements
 Nom du fichier généré
Les lignes de mouvements sélectionnées sont triées dans l'ordre suivant :
 Organisme
 Budget
 Exercice
 Mouvement
 Ligne de mouvement
7.2 	Structure du fichier texte généré
 Le fichier texte généré comporte une ligne par ligne de mouvement sélectionnée.
 Les données sont séparées par un point-virgule.
 Le nom du fichier texte dépend du paramètre passé en entrée du batch.
Champ 	Longueur 	Origine dans la base

Mouvement 	10C 	Mouvement.mouvement
